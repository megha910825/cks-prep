## Use Falco to Detect Threats

- How is falco installed in this cluster?
Falco is installed as a package on the controlplane node. You can run systemctl status falco to check the status.

- What is the status of the Falco service running on controlplane?
```
  systemctl status falco
```
- How can you check for the events generated by falco in this set up?
```
  journalctl -u falco-modern-bpf

```

- Alright, let us try that now. Open a new terminal window by clicking the + button next to the terminal. Then run journalctl -fu falco-modern-bpf.

The -f flag here keeps the journal logs open for the falco service and you can monitor any new logs that are generated.

- Now, let us update this rule by making minor changes to the way the output is logged.


Change the output so that it now prints the events in the following sample format as shown below:

Error Sensitive File Read Detected (user_loginuid=-1 command=cat /etc/shadow file=/etc/shadow container_name=simple-webapp-1)


To update, edit the rule in /etc/falco/falco_rules.local.yaml. Then reload the Falco configuration and restart the engine without restarting the service using kill -1 $(cat /var/run/falco.pid).

The misbehaving pod is still running, so the new output format should appear in the falco logs shortly after reloading.

Update the below rule in /etc/falco/falco_rules.local.yaml:

- rule: Read sensitive file untrusted
  enabled: false
  override:
    enabled: replace

- list: sensitive_file_list
  items: [/etc/shadow]

- rule: Read Sensitive File in Container
  desc: Detect when sensitive files are read inside a container
  condition: >
    open_read and container and fd.name in (sensitive_file_list)
  output: >
    Sensitive File Read Detected
    (user_loginuid=%user.loginuid command=%proc.cmdline file=%fd.name container_name=%container.name)
  priority: ERROR
  tags: [filesystem, mitre_credential_access]


Next, hot-reload the falco configuration by running kill -1 $(cat /var/run/falco.pid)
